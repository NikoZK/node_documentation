Server Side Rendering and routers

Server Side Rendering vs Client Side Rendering

Server-Side Rendering (SSR) means the server first builds the HTML THEN sends it to the browser. Client-Side Rendering (CSR) means the browser receives a little HTML content and then JavaScript builds the page itself.

In this project i will show how i use SSR, how the server reads HTML files and combines them with shared components like the header and footer before sending the final page.

SSR advantages

  • Faster first page load
  • Better for SEO because search engines receive full HTML and can scan through it
  • No browser-side CORS problems when the server fetches data itself

CSR advantages

  • Can feel and be made more dynamic after loading
  • Full page reloads feels faster for the user
  • Moves more workload to the browser instead of the server

So SSR spends more resources on the server, while CSR spends more resources in the browser.

Server-Side Rendering with Routers and Templating engine

In this project the server uses Server-Side Rendering (SSR). That means the server builds the full HTML page before sending it to the browser.

The flow works like this: 

app.js
-> pagesRouter.js
-> pagesUtil.js
-> templatingEngine.js
-> header.html + page.html + footer.html

First, app.js sets up Express and registers the routers: 

import pagesRouter from './routers/pagesRouter.js'
import replRouter from './routers/replRouter.js'

app.use(pagesRouter)
app.use(replRouter)

Then pagesRouter.js handles the page routes and sends the finished HTML pages: 

import { Router } from 'express'
import { frontpagePage, aboutPage, contactPage } from '../util/pagesUtil.js'

const router = Router()
    router.get('/', (req, res) => {
    res.send(frontpagePage)
})

router.get('/about', (req, res) => {
    res.send(aboutPage)
})

The page variables like cssLinks come from pagesUtil.js, where page files are read and passed into the templating engine: 

const frontpage = readPage('./public/pages/frontend/frontend.html')
export const frontpagePage = constructPage(frontpage, {
    cssLinks: '<link rel="stylesheet" href="/pages/frontend/frontend.css">'
})

Finally, templatingEngine.js reads shared HTML files and combines them: 

export function constructPage(page, options = {}) {
    const header = readPage('./public/components/header/header.html')
    const footer = readPage('./public/components/footer/footer.html')

    return header
        .replace('$$DOCUMENT_TITLE$$', options.documentTitle || 'Online Node.js REPL')
        .replace('$$CSS_LINKS$$', options.cssLinks || '')
        + page
        + footer
}

export function readPage(path) {
    return fs.readFileSync(path).toString()
}

This means the server reads the files, combines header, page content and footer so we avoid doing it over and over, and sends one finished HTML page to the browser.

Synchronous and Asynchronous File Reading

In this project the files are read with fs.readFileSync(), which is synchronous. That means the server will wait until the file is fully read before continuing. 

export function readPage(path) {
    return fs.readFileSync(path).toString()
}

Synchronous code is simpler to write and understand, but it blocks/stops execution while the file is being read. Asynchronous file reading is better for larger applications because it does'nt block the sequence of events in the same way.
In small projects the synchronous way is fine, but in bigger applications too much synchronous file reading will hurt the efficiency.
