Client vs Server and environment variables

NPM Scripts

In package.json we can define scripts that can be run with npm.
This makes it easier to start the project in different environments. 

{
  "scripts": {
    "start-dev": "cross-env PORT=8080 NODE_ENV=DEV nodemon.cmd app.js",
    "start-prod": "cross-env PORT=80 node app.js"
  }
}

These scripts can be run from the terminal with the commands: npm run start-dev or npm run start-prod
Using scripts makes project commands easier to remember and reuse.

Environment Variables

Environment variables are values provided to the application from outside the code. They are commonly used for configuration such as ports, API keys or the environment type. 

const PORT = process.env.PORT || 8080

const server = app.listen(PORT, (error) => {
    if (error) {
        console.log("Error starting the server")
    }
    console.log("Server is running on port", server.address().port, "The environment is:", process.env.NODE_ENV)
})

In Node.js we access environment variables with process.env. Here PORT decides which port the server runs on, while NODE_ENV can describe whether the app is running in development or production.

In this project we used cross-env in npm scripts, but the important part is reading values through process.env.
We can also manually set in in the terminal with $env:NODE_ENV = 'production' or $env:NODE_ENV = 'development' for windows at least.

Semantic HTML

Semantic HTML means using tags based on their meaning instead of just spamming <div> everywhere.
This makes the code easier to read and improves accessibility.

Examples of semantic tags:

  • <header>
  • <main>
  • <section>
  • <footer>
<header>
    <h1>My Website</h1>
</header>

<main>
    <section>
        <h2>Content</h2>
        <p>text text text</p>
    </section>
</main>

<footer>
    © 2026
</footer>

This makes the project easier to navigate in the long run, reuse code and maintain as it grows.

Client Files vs Server Files

In a Node.js project we separate client files and server files.

Server files run in Node.js and handle backend logic like routes and APIs.
Client files run in the browser and include files like HTML, CSS and the frontend specific JavaScript.

In this project, client files are placed inside the public folder:

  • assets/css - Styles
  • assets/js - Frontend JavaScript
  • assets/images - Images
  • pages - HTML pages

The server files like app.js is part of the backend and should not be exposed to the client.

Footer

We can keep the footer at the bottom of the page using flexbox on main and body: 

body {
    min-height: 95vh;
    display: flex;
    flex-direction: column;
}

main {
    flex: 1;
}

This makes the page take up the full height, and the main element fills all the space, pushing the footer down to the bottom.

CDN vs Local Files

A CDN (Content Delivery Network) allows us to load external libraries from the internet instead of storing them locally. 

<link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/water.css@2/out/water.css">

Pros

  • Easy to use and implement
  • No need to download any files
  • Can often be faster due to caching

Cons

  • Depends on internet connection
  • Relies on third-party maintenance
  • Less control compared to local files
